    - calcConsensus([{userName:'a', estimateValue:'5'}, {userName:'b', estimateValue:'3'}]) returns '5' (tie: pick first alphabetically among tied values)
    - calcConsensus([]) returns null
    - calcStats([{estimateValue:'1'}, {estimateValue:'3'}, {estimateValue:'5'}]) for numeric scheme returns { mode: '1', min: '1', max: '5', variance: '2.67' } (variance = population variance of numeric values, rounded to 2 decimals)
    - calcStats for non-numeric schemes (tshirt) returns { mode: 'M', min: null, max: null, variance: null } (only mode is meaningful)
    - generateCsv(rounds, playerNames) produces correct CSV string with headers and rows
    - CSV columns: Label, Consensus, Timestamp, Mode, Min, Max, Variance, then one column per player
    - downloadCsv triggers a file download (test that Blob is created correctly)
  </behavior>
  <action>
    **Create `planningpoker-web/src/utils/consensus.js`:**

    `calcConsensus(results)` — Takes array of `{userName, estimateValue}`. Returns the mode (most frequent estimateValue). On tie, pick the value that comes first alphabetically/numerically. Returns null if empty.

    `calcStats(results)` — Takes array of `{userName, estimateValue}`. Returns `{ mode, min, max, variance }`.
    - `mode`: same as calcConsensus result.
    - For numeric values (all estimateValues parse as numbers): `min` = smallest, `max` = largest, `variance` = population variance rounded to 2 decimal places.
    - For non-numeric values (T-shirt, custom text): `min`, `max`, `variance` are null. Only `mode` is populated.
    - Detection: try `Number(val)` — if all are finite numbers, treat as numeric.

    **Create `planningpoker-web/src/utils/csvExport.js`:**

    `generateCsv(rounds, playerNames)` — Takes array of round objects `{ label, consensus, votes: [{userName, estimateValue}], timestamp, mode, min, max, variance }` and a sorted array of all unique player names across all rounds. Returns CSV string.
    - Header row: `Label,Consensus,Timestamp,Mode,Min,Max,Variance,Player1,Player2,...`
    - Each round row: label (quoted if contains comma), consensus, ISO timestamp, mode, min or "", max or "", variance or "", then each player's vote for that round (or "" if they didn't vote in that round).
    - Properly escape CSV values: wrap in double quotes if value contains comma, quote, or newline. Escape internal quotes by doubling them.

    `downloadCsv(csvString, filename)` — Creates a Blob with `text/csv` type, creates an object URL, triggers download via a temporary `<a>` element with `download` attribute, then revokes the URL. Default filename: `planning-poker-export.csv`.

    **Write tests first (TDD):** Create `consensus.test.js` and `csvExport.test.js` using Vitest (already configured). Write failing tests, then implement.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Redux state -> CSV file | Round history in client memory exported as downloadable file |
| Vote data -> consensus calc | Untrusted vote values used in arithmetic |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-05 | Tampering | consensus.js | accept | Consensus is calculated from Redux state which comes from server-validated votes. Client-side override is host-only UI control with no security boundary (all data is visible to all session members already) |
| T-08-06 | Information Disclosure | CSV export | accept | CSV contains only data already visible to all session participants (votes, labels). Export is client-side only, no server endpoint. The file is saved to the user's local filesystem |
| T-08-07 | Injection | csvExport.js | mitigate | CSV values are properly escaped (double-quote wrapping, internal quote doubling) to prevent CSV injection. Values starting with =, +, -, @ are prefixed with a single quote to prevent formula injection in Excel/Sheets |
| T-08-08 | Denial of Service | reducer_rounds.js | accept | Round history is in-memory only, bounded by session lifetime. Typical sessions have 5-20 rounds. No persistence means no storage exhaustion risk |
</threat_model>
