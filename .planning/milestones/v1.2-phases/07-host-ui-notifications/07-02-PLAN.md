<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> POST /kick | Host sends kick request; backend validates host status |
| client -> POST /promote | Host sends promote request; backend validates host status |
| WS payload -> kick detection | Client infers kick from user list change |
| sessionStorage -> Welcome page | Kicked message passed via sessionStorage |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04 | Spoofing | kickUser/promoteUser action creators | mitigate | Backend validates caller is host via synchronized block (Phase 6 T-06-03). Frontend sends `currentUser` (from Redux) as `userName` param — cannot spoof another user's identity because backend checks session membership. |
| T-07-05 | Tampering | kick detection in PlayGame.jsx | accept | False positive possible if WS delivers stale user list during reconnect. Impact is low — user gets redirected to welcome and can rejoin. No data loss. |
| T-07-06 | Denial of Service | kick confirmation dialog bypass | mitigate | Kick requires confirmation dialog click. Direct API call (bypassing UI) is possible but backend authorization (host-only) limits abuse surface. |
| T-07-07 | Information Disclosure | sessionStorage kicked message | accept | Message is generic ("removed from session by host") — no sensitive data. Cleared on read. |
| T-07-08 | Elevation of Privilege | non-host calling kick/promote | mitigate | UI hides controls for non-host (isHost check). Backend returns 403 via HostActionException if non-host calls endpoint. Defense in depth. |
</threat_model>
