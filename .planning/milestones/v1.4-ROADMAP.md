## Phase Details

### Phase 9: Backend Concurrency & Cleanup
**Goal**: SessionManager is safe under concurrent load — no race conditions during session creation or eviction, and dead code is removed
**Depends on**: Phase 8
**Requirements**: CONC-01, CONC-02, CONC-03, CLN-02
**Success Criteria** (what must be TRUE):
  1. Concurrent session creation requests cannot produce duplicate session IDs (contains+add is atomic)
  2. A session being evicted cannot be observed in a partially-cleaned state by concurrent requests
  3. Burst messaging sends a consistent snapshot of results, not live map state that may change after the lock is released
  4. `getSessions()` method no longer exists in SessionManager
**Plans**: 2 plans

Plans:
- [x] 09-01-PLAN.md — SessionManager concurrency fixes (atomic createSession, eviction safety, dead code removal)
- [x] 09-02-PLAN.md — Burst messaging snapshot consistency

### Phase 10: Frontend Modernization
**Goal**: Frontend store setup uses current Redux patterns, the dead WebSocket subscription is gone, and lazy-loaded routes show a visible loading indicator
**Depends on**: Phase 9
**Requirements**: CLN-01, CLN-03, UI-04
**Success Criteria** (what must be TRUE):
  1. Redux store is created with `configureStore` (Redux Toolkit) — `createStore` and `redux-promise` are gone from the codebase
  2. No subscription to `/topic/items/${sessionId}` exists anywhere in the frontend source
  3. Navigating to a lazy-loaded route shows a visible loading indicator (spinner or skeleton) rather than a blank screen
**Plans**: 1 plan

Plans:
- [x] 10-01-PLAN.md — RTK configureStore migration, dead subscription verification, Suspense fallback verification
**UI hint**: yes

### Phase 11: UI Hardening & Test Coverage
**Goal**: Forms protect against double-submit, HTTP errors surface as non-blocking notifications, vote revert is complete, and the useStomp hook is covered by tests
**Depends on**: Phase 10
**Requirements**: UI-01, UI-02, UI-03, TEST-01
**Success Criteria** (what must be TRUE):
  1. Submitting JoinGame or CreateGame disables the form and shows a spinner until the request completes or fails
  2. An HTTP error from any action displays as a MUI Snackbar notification — no browser `alert()` dialogs remain
  3. A failed vote HTTP request resets both the voted flag and the optimistic result entry, leaving the UI in the pre-vote state
  4. `useStomp` hook has passing unit tests covering connect, subscribe, reconnect, and disconnect lifecycle events
**Plans**: 1 plan

Plans:
- [x] 11-01-PLAN.md — Verify SC-1/2/3 already met, add useStomp reconnect lifecycle tests
