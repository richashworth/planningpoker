# Planning Poker — Requirements

**Milestone:** v1.5 UX & Polish
**Status:** Defining
**Last updated:** 2026-04-10

## Goal

Improve session UX with deliberate label broadcast, accessible state transitions, and production-grade logging hygiene.

## v1.5 Requirements

### UX

- [x] **UX-01** — Host types round label locally in the TextField without broadcasting on every keystroke. Label broadcasts to all participants only when the host clicks a **Set** button or presses **Enter**. Replaces the current 300ms debounced live-typing dispatch at `planningpoker-web/src/containers/Vote.jsx:64-71`. *(completed 2026-04-10, plan 12-01)*
  - Empty submissions are allowed (clears the label)
  - Set button is disabled while input matches the last broadcast value (no-op prevention)
  - Non-host participants see the read-only label exactly as before

### Accessibility

- [x] **A11Y-01** — Vote reveal transition announces to screen readers via an `aria-live="polite"` region. Announcement includes total vote count (e.g., "Votes revealed: 5 of 5 players voted"). *(completed 2026-04-10, plan 12-02)*
  - Region is visually hidden but present in the DOM
  - Announcement fires once per reveal, not per result payload burst

- [x] **A11Y-02** — Consensus result announces via `aria-live="polite"` region. Fires on auto-majority detection and on host override. *(completed 2026-04-10, plan 12-02)*
  - Announcement includes the consensus value (e.g., "Consensus: 5")
  - Host override re-announces with the new value
  - Re-announcements are debounced to avoid spam during rapid changes

### Logging

- [x] **LOG-01** — Audit all `logger.info()` and `logger.debug()` calls across `planningpoker-api/src/main/java`. Downgrade hot-path logs (burst messaging in `MessagingUtils`, vote dispatch in `VoteController`, per-message WebSocket activity) from INFO to DEBUG so production logs capture lifecycle events, not per-interaction noise. *(completed 2026-04-10, plan 13-01)*

- [x] **LOG-02** — Scrub PII from all log statements. No session IDs, usernames, or vote values appear in log output at any level. Use log-safe identifiers (e.g., session hash prefix, user count) where diagnostics still need correlation. *(completed 2026-04-10, plan 13-01 — LogSafeIds SHA-256 masking helper + LoggingHygieneTest regression guard)*

- [x] **LOG-03** — Set an appropriate production log level in `planningpoker-api/src/main/resources/application.properties`. Root level INFO; application package at INFO; hot-path classes at WARN or configurable via env. Verify Logback/Spring Boot config honors the setting in the fat JAR. *(completed 2026-04-10, plan 13-02 — env-var overrides: `LOG_LEVEL_ROOT`, `LOG_LEVEL_SPRING`, `LOG_LEVEL_APP`, `LOG_LEVEL_HOTPATH`)*

## Out of Scope (v1.5)

- Broader accessibility audit (keyboard nav, focus order, contrast) — separate milestone
- Structured logging / JSON log format — not needed at current scale
- Log aggregation or shipping — Railway captures stdout
- Mid-session label history UI — labels already persist in round history
- Localized announcements — English only

## Success Criteria

- Typing in the label field no longer dispatches WebSocket messages; only Set/Enter does
- Screen reader users hear vote reveal and consensus announcements at appropriate moments
- Production logs at INFO level contain no session IDs, usernames, or vote values
- All existing e2e tests still pass; new tests cover Set button and Enter key behavior
