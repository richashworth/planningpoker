       ```java
       @PostMapping("kick")
       public void kickUser(
               @RequestParam(name = "userName") final String userName,
               @RequestParam(name = "targetUser") final String targetUser,
               @RequestParam(name = "sessionId") final String sessionId
       ) {
           synchronized (sessionManager) {
               validateSessionMembership(sessionId, userName);
               if (userName.equalsIgnoreCase(targetUser)) {
                   throw new IllegalArgumentException("cannot kick yourself");
               }
               if (!userName.equalsIgnoreCase(sessionManager.getHost(sessionId))) {
                   throw new HostActionException("only the host can perform this action");
               }
               if (!containsIgnoreCase(sessionManager.getSessionUsers(sessionId), targetUser)) {
                   throw new IllegalArgumentException("target user is not a member of this session");
               }
               sessionManager.removeUser(targetUser, sessionId);
               logger.info("{} has kicked {} from session {}", userName, targetUser, sessionId);
           }
           messagingUtils.burstUsersMessages(sessionId);
           messagingUtils.burstResultsMessages(sessionId);
       }
       ```

       Key design per decisions:
       - Per D-01: Non-host throws `HostActionException` with message "only the host can perform this action" -> 403
       - Per D-02: Host check (`getHost`) is inside `synchronized (sessionManager)` block, race-safe
       - Per D-03: Uses existing `removeUser()` which clears estimates (same as voluntary leave)
       - Per D-04: No ban list -- kicked user can rejoin normally
       - Per NOTIF-02: `burstUsersMessages` pushes updated user list + host to all WebSocket subscribers

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /kick endpoint | Untrusted caller claims to be host via userName parameter |
| client -> /promote endpoint | Untrusted caller claims to be host via userName parameter |
| userName/targetUser params -> SessionManager | User-supplied strings used for session state mutation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-03 | Spoofing | kickUser userName param | mitigate | Host identity is verified server-side: `getHost(sessionId)` compared to `userName` param inside synchronized block. Attacker cannot spoof by sending someone else's name because the check uses server state, not client claims alone. NOTE: The app has no authentication -- users self-identify by name. This is an accepted architectural constraint (no auth per CLAUDE.md). The host check prevents non-host names from executing host actions. |
| T-06-04 | Elevation of Privilege | Non-host calling /kick or /promote | mitigate | `HostActionException` thrown (403) if caller's userName does not match `sessionManager.getHost(sessionId)`. Check is inside `synchronized (sessionManager)` per D-02 to prevent TOCTOU race. |
| T-06-05 | Information Disclosure | Error messages | accept | Error messages ("only the host can perform this action", "target user is not a member") reveal session structure. Acceptable for a casual planning poker app with no auth. |
| T-06-06 | Tampering | IDOR - kick/promote with wrong sessionId | mitigate | `validateSessionMembership(sessionId, userName)` verifies both that the session exists AND that the caller is a member. A user cannot kick someone in a session they haven't joined. |
| T-06-07 | Denial of Service | Rapid kick requests | accept | No rate limiting. Acceptable for internal team tool. Kicked users can rejoin per D-04, limiting impact. |
</threat_model>
